Critical Performance Issue

2-Second Blocking Sleep (config.lua:123)

When user presses Ctrl+Shift+E to open scrollback in nvim, the UI thread blocks for 2 full seconds while waiting for nvim to read a temp file.

config.lua:92-125
wezterm.on("trigger-nvim-with-scrollback", function(window, pane)
  local scrollback = pane:get_lines_as_text(2000)
  local name = os.tmpname()
  local f = io.open(name, "w+")
  f:write(scrollback); f:flush(); f:close()
  
  window:perform_action(SpawnCommandInNewWindow, pane)
  
  wezterm.sleep_ms(2000)  -- ⚠️ BLOCKS UI for 2 seconds
  os.remove(name)
end)

Why it exists:

SpawnCommandInNewWindow is asynchronous. Need to ensure nvim opens the temp file before deleting it. No way to await spawn completion.

Impact:

  • UI completely frozen for 2 seconds
  • User cannot interact with WezTerm during this time
  • Only triggered manually (not automatic)

Optimization Opportunities (Priority Order)
High Priority

Reduce Nvim Handler Sleep

Reduce wezterm.sleep_ms(2000) to wezterm.sleep_ms(100)

Impact: High - eliminates 1.9 seconds of UI freeze. Risk: Low - 100ms sufficient for file open.

Medium Priority

Optimize Hyperlink Array Construction

Single-pass building instead of double iteration (lines 68-70, then 73-78)

Impact: Low - ~1-2ms at startup. Risk: None - improves correctness.

Low Priority

Remove unused fallback fonts (8 fonts may be excessive)

Impact: Low - maybe 5-10ms improvement. Risk: Low - requires glyph usage testing.

Overall Verdict: This WezTerm configuration is already highly optimized and follows performance best practices. The only significant issue is the 2-second blocking sleep in the nvim scrollback handler (config.lua:123), which can be easily fixed by reducing the sleep duration to 100ms. All other optimizations are minor and provide marginal improvements. The configuration demonstrates excellent understanding of WezTerm APIs, strong software engineering practices, and performance-conscious design.
